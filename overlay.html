System design Overlay desktop app · Windows

Overlay App that makes meetings, classes and everyday work faster.

This is the system design of my Overlay App – a top-most Windows utility with WebView2, global hotkeys, auto-typing, quick paste and robust screenshot features. It helps during online meetings, coding, chats and exams by keeping common actions always accessible.

Built in C++ · Win32 · WebView2
Use case: meetings, classes, demo sessions
Features: overlay, hotkeys, automation

One goal: reduce context switching and make repetitive actions (paste, type, screenshot) as fast as a single key.

Problem & Goals

Switching between windows just to paste text, type long answers, or capture screens wastes time during live sessions.

🎯 Always-on overlay that sits above apps.
⌨️ Global hotkeys for toggle, pause, opacity.
🤖 Auto typing from local buffer / clipboard.
🖼️ Screenshot of active window to clipboard.
Designed for real use in meetings & classes. Implementation: single Windows process.

Architecture overview

The Overlay App runs as a single Windows desktop process with several internal subsystems: overlay window manager, WebView2 panel, control panel UI, hotkey manager, automation engine, and screenshot / clipboard services.

Key design choices

The app is intentionally kept as a single lightweight process with clear subsystems instead of multiple services:

  • Overlay window is WS_EX_TOPMOST and can be NOACTIVATE for click-through mode.
  • Left pane hosts WebView2 (for Poe / WhatsApp).
  • Right pane is a control panel with buttons, sliders, and a fake input preview.
  • Global keyboard hook + hotkeys coordinate automation.
  • Automation engine chooses between JS injection and OS typing.

This keeps the system fast, responsive and focused on helping in real-time sessions.

Components
Main subsystems

Each part of the Overlay App has a clear responsibility. This section is structured exactly how you can explain in a system design / project interview.

1. Overlay Window Manager

  • Creates main WS_POPUP window with WS_EX_TOPMOST | WS_EX_TOOLWINDOW | WS_EX_NOACTIVATE.
  • Splits client area into WebView pane (left ~80%) and control panel (right ~20%).
  • Handles WM_SIZE to resize WebView2 controller.
  • Custom painting via WM_ERASEBKGND and WM_CTLCOLORBTN for theme.
  • Opacity slider uses SetLayeredWindowAttributes.
  • WM_NCHITTEST returns HTTRANSPARENT over Web area in overlay mode so clicks go to underlying app.

2. Web Panel (WebView2 Host)

  • Hosts WebView2 as the left panel of the overlay window.
  • Default navigation to Poe.com; “Switch” button toggles to WhatsApp Web.
  • Exposes current URL via get_Source so the automation engine can inspect the host.
  • Only allows JS injection on trusted hosts (Poe, Bing, Copilot, HuggingFace, localhost…).

3. Control Panel UI

  • Right-side panel with Win32 buttons and sliders.
  • Buttons: Copy, Paste, Screenshot, Run typing, Auto typing, PasteOnly, Theme toggle, Activate, Switch.
  • Speed slider: character typing delay (10–200 ms).
  • Opacity slider: overlay transparency (50–255).
  • “Fake input” static control: focus-free view of the local text buffer.
  • All wired via WM_COMMAND handlers.

4. Input & Hotkey Manager

  • Global hotkeys via RegisterHotKey:
  • Alt+Q: double-tap to show/hide overlay.
  • Alt+P: pause/resume typing.
  • Alt+E / Alt+R: adjust opacity.
  • Low-level keyboard hook (WH_KEYBOARD_LL):
  • F9 triggers typing and is swallowed (no leak to apps).
  • While overlay visible, non-ctrl/alt keystrokes go into local fake buffer (including Backspace, Enter, Tab).

5. Automation Engine

  • Core logic for auto typing & injection.
  • RunTyping(): OS-level typewriter using SendInput.
  • Hides overlay before typing using RobustHideForFocus.
  • Uses local buffer first; if empty, falls back to clipboard text.
  • AutoTypeFromClipboard():
  • Decides between JS injection and OS typing based on host.
  • JS uses “Backspace wake” trick to enable send buttons.

6. Clipboard & Screenshot Service

  • Clipboard helper reads CF_UNICODETEXT into std::wstring.
  • Used by OS typing, Auto typing, and Paste-only.
  • RobustScreenshotToClipboard():
  • Hides overlay, targets GetForegroundWindow().
  • Captures using PrintWindow(PW_RENDERFULLCONTENT), fallback to BitBlt.
  • Places bitmap as CF_BITMAP on clipboard.

Flows
Key user journeys

These flows are explained exactly like interview answers: clear triggers, components involved, and what happens step by step.

Flow 1 – User presses F9 for auto typing

  1. User presses F9.
  2. Low-level keyboard hook detects F9, sets g_startTyping = true and returns 1 (so active app never sees F9).
  3. Main message loop calls CheckAndRunTyping() after each message.
  4. It sees g_startTyping, resets it and spawns a RunTyping() thread.
  5. RunTyping() hides overlay, picks text from local buffer or clipboard, simulates keystrokes using SendInput.
  6. Typing respects pause state; delay between chars is read from speed slider.
  7. At the end, overlay is shown again.

Flow 2 – User clicks “Auto” button

  1. User presses Auto on control panel.
  2. WM_COMMAND dispatches to AutoTypeFromClipboard().
  3. Engine reads text from local buffer (preferred) or clipboard.
  4. Gets current host from WebView2 and checks if injection is allowed (CanInjectForHost).
  5. If allowed: builds Backspace-wake JS via BuildInjectionJS_BackspaceWake() and executes it.
  6. JS fills textarea / textbox, triggers input/change events, then does the backspace + restore trick to wake “Send” buttons.
  7. If result is not true or host not allowed: fallback to OS-level RunTyping().

Flow 3 – Screenshot current window

  1. User clicks SS button on control panel.
  2. Calls OnSsClicked() → RobustScreenshotToClipboard().
  3. RobustHideForFocus() hides overlay and waits until some other window becomes foreground.
  4. Foreground window is captured using DwmGetWindowAttribute + PrintWindow.
  5. If PrintWindow fails, fallback to screen copy with BitBlt.
  6. Bitmap is placed into clipboard as CF_BITMAP.
  7. Overlay is shown again for next actions.

Flow 4 – Toggle overlay vs interactive mode

  1. User clicks Activate / Deactivate button.
  2. ToggleActivationMode() flips internal flag g_isInteractive.
  3. Window is temporarily hidden, WS_EX_NOACTIVATE is added/removed via SetWindowLong.
  4. SetWindowPos(..., SWP_FRAMECHANGED) forces Windows to re-evaluate activation rules.
  5. Window is shown again.
  6. In overlay mode: web pane is click-through, app never steals focus.
  7. In interactive mode: behaves like normal topmost app for configuration.
